# AGENTS.md

Guidance for coding agents working in this repository.

## Project Purpose

No Caves - Continued is a small RimWorld 1.6 Harmony mod. Its purpose is to prevent vanilla cave tunnel generation during map generation.

The current implementation patches RimWorld's cave-generation utility with a Harmony prefix that returns `false`, which skips the original method. It also preserves the original NoCaves behavior by forcing RimWorld's vanilla cave-map check to return false.

The RimWorld 1.6 targets are:

```text
RimWorld.MapGenCavesUtility.GenerateCaves
RimWorld.Planet.World.HasCaves
```

Do not describe or implement this as a universal blocker for every modded cave system. The maintained behavior is narrower: patch vanilla cave generation, suppress vanilla cave-map checks, and leave independent mod generators alone unless the user explicitly asks for broader behavior.

Known compatibility context: Geological Landforms appeared compatible during testing. Its Surface Caves and Directional Caves are generated by Geological Landforms itself and still generate with this mod enabled. The README should continue to communicate that only vanilla cave generation is patched specifically.

## Repository Layout

```text
.github/workflows/release.yml
  Main-branch release workflow. Builds on ubuntu-latest, packages the mod, validates the zip, and creates a GitHub release.

.gitignore
  Keeps local OS files, IDE files, .NET intermediates, Python caches, logs, and locally generated archives out of git.

About/About.xml
  RimWorld mod metadata, package id, supported game version, Harmony dependency, and load order.

LoadFolders.xml
  RimWorld version folder mapping. For RimWorld 1.6, load the repo root and the 1.6 folder.

1.6/Assemblies/
  Packaged mod assembly output consumed by RimWorld.

Source/NoCavesContinued.csproj
  .NET project file. Builds directly to ../1.6/Assemblies/.

Source/NoCavesContinued/NoCavesContinued.cs
  The full runtime implementation.

scripts/publish-steam-workshop.sh
  Local helper for staging the latest GitHub release zip and publishing it with SteamCMD.

scripts/generate-steam-workshop-description.py
  Converts Steam-facing README sections from Markdown to Steam BBCode.

Source/obj/
  Generated build intermediates. Do not edit by hand.
```

`SteamWorkshopDescription.txt` is generated into release packages. It is not a source file in the repository unless the user intentionally decides to commit a static copy.

## Prerequisites

- Git.
- .NET SDK 8.0 or newer.
- Network access for `dotnet restore`.
- RimWorld 1.6 and Harmony for runtime verification.
- `zip` and `unzip` only if validating release packages locally; normal local builds do not need them.
- `gh`, `unzip`, and SteamCMD for local Steam Workshop publishing with `scripts/publish-steam-workshop.sh`.
- Python 3.10+ and `md2steam==1.0.1` for local Steam Workshop description generation.

The project targets `net472`, but the required .NET Framework reference assemblies are restored from NuGet. Do not require a Windows runner or local .NET Framework install for ordinary builds.

## Build

Build from the `Source` directory:

```bash
dotnet restore
dotnet build -c Release
```

Expected output:

```text
1.6/Assemblies/NoCavesContinued.dll
1.6/Assemblies/NoCavesContinued.pdb
```

The project targets `net472` for RimWorld/Unity compatibility. Keep the build output path aligned with RimWorld's mod folder layout unless intentionally changing packaging.

## README And Steam Description

`README.md` is dual-purpose: it is the GitHub README and the source text for the Steam Workshop page. Steam Workshop users are the primary audience for the top half of the README, so keep player-facing install, behavior, credit, save-safety, and compatibility notes before development instructions.

The release workflow generates `SteamWorkshopDescription.txt` from `README.md` with:

```bash
scripts/generate-steam-workshop-description.py README.md "$RUNNER_TEMP/SteamWorkshopDescription.txt"
```

The generator uses `md2steam==1.0.1` to convert Markdown into Steam's BBCode-style markup, such as `[h1]`, `[h2]`, `[url=...]`, `[list]`, `[olist]`, and `[code]`. Do not hand-roll Markdown-to-Steam-BBCode conversion unless `md2steam` stops meeting the repository's needs.

By default, the generator excludes GitHub/developer-only sections from the Workshop description:

```text
Manual Install
GitHub And Development
Development Prerequisites
Build
Development Test
Steam Workshop Publishing
```

Use `--include-dev-sections` only when intentionally converting the full README.

## Verification

At minimum, run:

```bash
dotnet build Source/NoCavesContinued.csproj -c Release
```

There is no automated test harness in this repository. Runtime verification requires RimWorld:

1. Install the entire `NoCavesContinued` folder as a RimWorld mod.
2. Load Harmony before this mod.
3. Start a new map on a mountainous or impassable tile that would normally contain caves.
4. Confirm the RimWorld log contains messages like:

```text
[No Caves - Continued] Patched RimWorld.MapGenCavesUtility.GenerateCaves(...); cave generation will be skipped.
[No Caves - Continued] Finished patching 1 cave-generation method(s).
[No Caves - Continued] Patched RimWorld.Planet.World.HasCaves(...); cave checks will report false.
[No Caves - Continued] Finished patching 1 cave-check method(s).
```

If patching fails, inspect the diagnostic log emitted by the mod. It lists loaded cave-related types and declared method names to help identify a changed RimWorld target.

For release-package verification, the expected zip shape is:

```text
NoCavesContinued/About/About.xml
NoCavesContinued/About/Preview.png
NoCavesContinued/LoadFolders.xml
NoCavesContinued/README.md
NoCavesContinued/SteamWorkshopDescription.txt
NoCavesContinued/1.6/Assemblies/NoCavesContinued.dll
NoCavesContinued/1.6/Assemblies/NoCavesContinued.pdb
```

The release zip should not include `.git`, `.github`, `Source`, or `AGENTS.md`.

## Implementation Notes

- `Bootstrap` is marked with `Verse.StaticConstructorOnStartup`, so RimWorld runs its static constructor when the mod assembly loads.
- The Harmony ID is `sayhiben.nocavescontinued`; keep it stable unless intentionally creating a distinct mod identity.
- Patch targets are resolved by string name through `HarmonyLib.AccessTools`. This avoids compile-time references to RimWorld internal types that may be absent from public reference assemblies.
- The cave-generation target is `RimWorld.MapGenCavesUtility.GenerateCaves`.
- The vanilla cave-map check target is `RimWorld.Planet.World.HasCaves`.
- The patches intentionally apply to every declared overload matching their target method names.
- The prefix methods should stay minimal. Returning `false` is the behavior that skips cave generation and cave checks; `ReportNoCaves` must also set `__result = false`.
- Keep the failure diagnostics useful. If targets change, preserve or improve the cave-type discovery output instead of replacing it with a generic error.

## GitHub Release Workflow

The release workflow runs on pushes to `main`. It intentionally uses `ubuntu-latest`; the project builds there because `Microsoft.NETFramework.ReferenceAssemblies.net472` is restored from NuGet.

The workflow:

1. Checks out the repo.
2. Sets up .NET 8.
3. Restores and builds `Source/NoCavesContinued.csproj` in Release.
4. Creates a temporary Python venv and installs `md2steam==1.0.1`.
5. Generates `SteamWorkshopDescription.txt` from `README.md`.
6. Packages only the distributable RimWorld mod folder.
7. Validates required and forbidden paths.
8. Creates a GitHub release tagged `build-${GITHUB_RUN_NUMBER}` with `NoCavesContinued-${GITHUB_RUN_NUMBER}.zip`.

If editing packaging, keep the validation list in sync with the intended distributable contents. The GitHub release zip is the source package used by the local Steam Workshop helper.

## Steam Workshop Publishing

Do not put Steam credentials in GitHub Actions. Workshop publishing should remain local/manual unless the user explicitly chooses a secured publishing setup.

The first Steam Workshop upload must be done manually through RimWorld/Steam so Steam creates a real Workshop item ID. The local helper updates an existing item only and refuses `publishedfileid=0`.

After the first item exists, use:

```bash
STEAM_USERNAME=your_steam_username \
scripts/publish-steam-workshop.sh \
  --published-file-id 1234567890 \
  --changenote "Update from latest tested GitHub release" \
  --dry-run
```

Remove `--dry-run` after inspecting the generated VDF. The helper downloads the latest GitHub release zip with `gh`, stages the mod, writes `About/PublishedFileId.txt` into the staged copy, uses `SteamWorkshopDescription.txt` from the release by default, and calls SteamCMD. It intentionally does not accept or store a Steam password; SteamCMD should prompt or use local trusted login/Steam Guard state.

When writing SteamCMD VDF metadata, preserve real newline characters inside multiline values. Do not convert description newlines to `\n`; Steam Workshop renders those sequences literally on the page.

## Editing Guidelines

- Keep changes narrowly scoped. This mod has one job: skip vanilla cave generation.
- Prefer compatibility with RimWorld 1.6 and Unity's Mono runtime over newer .NET idioms.
- Do not add runtime copies of Harmony, RimWorld, Unity, or reference assemblies to the mod package. Harmony is a declared mod dependency.
- Do not edit generated files under `Source/obj/`.
- If source code changes affect runtime behavior, rebuild so `1.6/Assemblies/NoCavesContinued.dll` matches the source.
- Preserve the simple C# style already present: block-scoped namespace, explicit helper methods, and clear log messages.
- Avoid unrelated formatting churn in XML or C# files.
- Use concise comments only where they explain RimWorld/Harmony behavior that is not obvious from the code.
- Do not add Steam credentials to GitHub Actions. Steam Workshop publishing should stay local/manual unless the user explicitly chooses a secured publishing setup.
- Do not hand-roll Markdown-to-Steam-BBCode conversion unless `md2steam` stops meeting the repository's needs.
- Keep `About/Preview.png` valid for Steam/RimWorld publishing; RimWorld recommends a 640x360 PNG under 1 MB.

## Release Checklist

Before considering a change complete:

1. Confirm `About/About.xml` still declares the correct package id, supported RimWorld version, Harmony dependency, and load order.
2. Confirm `LoadFolders.xml` still maps RimWorld 1.6 to the root and `1.6` folders.
3. Run `dotnet build -c Release` from `Source`.
4. Confirm the built DLL is present under `1.6/Assemblies/`.
5. For behavior changes, verify in RimWorld using a cave-prone map tile and inspect the game log.
6. If the README changed, generate the Steam description locally or rely on the workflow and inspect the release artifact before publishing to Steam.
